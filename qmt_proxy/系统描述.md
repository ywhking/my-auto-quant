# QMT Proxy System Description

## Overview

The QMT Proxy acts as a trading message relay between the Trading Initiator and Trading Executor. It receives trading instructions from the initiator, forwards them to the executor, and returns execution results back.

## Architecture

```
┌─────────────────┐         ┌──────────────┐         ┌─────────────────┐
│ Trading         │         │              │         │ Trading         │
│ Initiator       │◄────────┤   QMT Proxy  ├────────►│ Executor       │
│ (Northbound)    │ WebSocket│   (This)     │ WebSocket│ (Southbound)   │
└─────────────────┘         └──────────────┘         └─────────────────┘
```

## Main Features

1. **Authentication**: Token-based authentication for WebSocket connections
2. **Message Forwarding**: Bidirectional message routing between initiator and executor
3. **Connection Management**: Single WebSocket endpoint with role-based routing
4. **Error Handling**: Immediate rejection if executor connection is unavailable

## API Interfaces

### 1. Get Authentication Token

**Endpoint:** `GET /auth`

**Description:** Authenticates user credentials and returns a token for WebSocket connection.

**Query Parameters:**
- `userName`: Username (e.g., `initiator` or `executor`)
- `password`: User password

**Response:**
```
200 OK
{
    "token": "abc123-def456-ghi789"
}
```

**Error Response:**
```
401 Unauthorized
{
    "error": "Invalid credentials"
}
```

**Configuration File (`config/users.json`):**
```json
{
    "users": [
        {
            "name": "initiator",
            "password": "xxxxxxx"
        },
        {
            "name": "executor",
            "password": "xxxxxxx"
        }
    ]
}
```

---

### 2. WebSocket Connection

**Endpoint:** `GET /ws?token=abc123-def456-ghi789`

**Description:** Establishes WebSocket connection. The connection role is determined by the authenticated user.

#### Northbound Connection (Trading Initiator)

Used by the Trading Initiator to:
1. Send trading messages to be forwarded to the executor
2. Receive execution results from the executor

**Message Format (Initiator → Proxy → Executor):**
```json
{
    "stock": "000001.SH",
    "action": "buy",
    "price": 20,
    "number": 1000
}
```

**Response Format (Executor → Proxy → Initiator):**
```json
{
    "status": "success",
    "data": {
        "stock": "000001.SH",
        "action": "buy",
        "price": 19.60,
        "number": 1000
    }
}
```

#### Southbound Connection (Trading Executor)

Used by the Trading Executor to:
1. Receive trading messages from the initiator
2. Send execution results back to the initiator

**Message Format (Proxy → Executor):**
```json
{
    "stock": "000001.SH",
    "action": "buy",
    "price": 20,
    "number": 1000
}
```

**Response Format (Executor → Proxy):**
```json
{
    "status": "success" | "error",
    "data": {
        "stock": "000001.SH",
        "action": "buy",
        "price": 19.60,
        "number": 1000
    }
}
```

## Connection Flow

### Normal Operation

1. **Initiator** requests token from `/auth?userName=initiator&password=xxxxxxx`
2. **Initiator** connects to `/ws?token=<token>` (role: initiator)
3. **Executor** requests token from `/auth?userName=executor&password=xxxxxxx`
4. **Executor** connects to `/ws?token=<token>` (role: executor)
5. **Initiator** sends trading message via WebSocket
6. **Proxy** forwards message to Executor's WebSocket connection
7. **Executor** processes order via QMT client
8. **Executor** sends result via WebSocket
9. **Proxy** forwards result to Initiator's WebSocket connection

### Error Handling

**If Executor Connection is Down:**
- Proxy rejects trading messages from Initiator immediately
- Returns error response:
```json
{
    "status": "error",
    "message": "Executor not connected"
}
```

**If Initiator Connection Fails:**
- WebSocket connection is closed
- No message queuing (real-time only)

## Message Data Types

### Trading Message
```json
{
    "stock": "string",      // Stock code (e.g., "000001.SH")
    "action": "buy|sell",   // Trading action
    "price": "number",       // Limit price
    "number": "number"      // Quantity
}
```

### Execution Result
```json
{
    "status": "success|error",
    "data": {
        "stock": "string",
        "action": "buy|sell",
        "price": "number",      // Actual executed price
        "number": "number"       // Actual executed quantity
    },
    "message": "string"  // Error message (if status is error)
}
```

## Implementation Notes

1. **Token Storage**: Tokens stored in memory with user role mapping
2. **Connection Mapping**: Maintain mapping of active connections by role
3. **Message Routing**: Forward messages between initiator and executor connections
4. **Connection Timeout**: Implement WebSocket ping/pong for connection health
5. **Logging**: Log all message forwarding for audit trail
